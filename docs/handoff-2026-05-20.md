## 二、工具站已实现功能

首页中文首屏可见：

```text
不用拖拽 40 万行，生成 Excel 批量序列
```

真实互动工具位于首页：

```text
/#generator
```

工具支持输入：

- 前缀：如 `Account`、`SKU`、`Batch`
- 起始数字
- 生成行数
- 步长
- 分隔符
- 补零位数

工具输出：

- Excel 365 `SEQUENCE` 公式
- 旧版 Excel `ROW` 公式
- 纯文本预览
- CSV 预览

交互能力：

- 快速示例：`Account1...`、`SKU-0001...`、`Batch 100...`
- 重置
- 复制按钮
- 复制后显示“已复制”
- 大行数只预览前 100 行，避免浏览器卡顿
- 错误状态：空前缀、非法行数、步长为 0、非法分隔符、非法补零位数

核心规则：

- v1 是确定性工具，不接 AI。
- v1 不需要登录、支付、积分、数据库持久化、文件上传。
- 不编辑用户 Excel 文件，只生成可复制公式和预览文本。

## 六、已知 caveats

### 1. 普通 `next build` 在 Windows 可能遇到 standalone symlink 问题

模板 `next.config.mjs` 默认：

```js
output: process.env.VERCEL ? undefined : 'standalone'
```

在 Windows 非管理员环境中，普通：

```bash
pnpm exec next build
```

可能在 standalone 复制依赖阶段遇到：

```text
EPERM: operation not permitted, symlink ...
```

当前推荐用：

```powershell
$env:VERCEL='1'
pnpm exec next build
```

这可以区分“应用代码是否能构建”和“本机 standalone 打包权限问题”。

### 2. `next.config.mjs` 有模板级 warning

构建时会提示：

```text
Unrecognized key(s) in object: 'turbopackFileSystemCacheForDev' at "experimental"
Unrecognized key(s) in object: 'reactCompiler'
```

这是当前模板配置与 Next 15.5.7 的兼容 warning，不是本次工具实现引入。当前不影响构建通过。

### 3. 不要同时跑 dev server 和 build

本地曾出现过 dev server 与 build 同时争用 `.next`，导致 manifest/cache 异常或 build 卡住。

推荐流程：

```powershell
# 先停止 3001 dev server
Get-NetTCPConnection -LocalPort 3001 -ErrorAction SilentlyContinue

# 确认没有本项目 next build / dev 残留进程
Get-CimInstance Win32_Process | Where-Object { $_.CommandLine -like '*D:\toolsites\excel-sequence-generator*' }

# 必要时删除 .next
Remove-Item -LiteralPath 'D:\toolsites\excel-sequence-generator\.next' -Recurse -Force
```

然后再单独 build 或 dev。
